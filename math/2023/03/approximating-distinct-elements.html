Approximating Distinct Element in a Stream

This post explains a probabilistic counting algorithm with which one can estimate the number of distinct elements in a large collection of data in a single pass.

probability, algorithm, data-science, machine-learning, bloom-filter, online-learning

Published

08 March 2023

This post is based on notes by Alex Tsun and paper by Flajolet and Martin (1985). It is pretty amazing the the algorithm discovered in the 1980s when the memory was very expensive has a huge number of applications in networking, database, and computer system now - big data era (Flajolet & Martin, 1985).

Motivation problem

For one of the most popular videos on Youtube, let’s say there are N=2N = 2N=2 billion views, with n=900n = 900n=900 million of them being distinct views. How much space is required to accurately track this number? Well, let’s assume a user ID is an 8-byte integer. Then, we need 900,000,000×8900,000,000 \times 8900,000,000×8 bytes total if we use a Set to track the user IDs, which requires 7.27.27.2 gigabytes of memory for ONE video.

Intuitions and examples

When we count distinct values nnn, we notice that it has the following property:

  • n>=0n >= 0n>=0 for sure
  • nnn is monotonic increasing

For instance, we cannot have −2-2−2 distinct values (distinct users). Furthermore, the distinct values always increase or remain constant, such as we have 777 distinct users at the current moment and this value will either increase or remain constant during the streaming process.

Instead of having a set of distinct values and querying the new value is unique or not, we could rely on hashing and probability to count the number of distinct values with O(1)O(1)O(1) space.

Figure 1. Illustration of using hash function to count the number of distinct values.

The idea of approximating distinct elements in a stream is to use the minimum value generated randomly from the hash function as it is illustrated in the Figure 1.

Probabilistic reasoning

If U1,⋯ ,Um∼Unif(0,1)U_1, \cdots, U_m \sim \mathrm{Unif}(0, 1)U1​,⋯,Um​∼Unif(0,1) (continuous) are iid, and X=min{U1,⋯ ,Um}X = \mathrm{min}\{U_1, \cdots, U_m \}X=min{U1​,⋯,Um​} is their minimum where mmm is the number of distinct values, then

E[X]=1m+1(1) \mathbb{E}[X] = \frac{1}{m+1} \tag{1} E[X]=m+11​(1)

To prove the above statement, we start working with probabilities first (e.g., the CDF FX(x)=P(X≤x)F_X(x) = \mathbb{P}(X \leq x)FX​(x)=P(X≤x)) and take the derivative to find the PDF (this is a common strategy for dealing with continuous random variables).

For the second equation, we use the fact that the minimum of numbers is greater than a value if and only if all of them are.

Now, take the derivative of Equation (3), we can have

Median trick

In equation (11), the space complexity for the hash function grows 20 times if we set our failure rate δ=0.05\delta = 0.05δ=0.05,

k=10.05×ϵ2=20ϵ2 k = \frac{1}{0.05 \times \epsilon^2} = \frac{20}{\epsilon^2} k=0.05×ϵ21​=ϵ220​

If we want to reduce our space complexity, we could use the “median-of-means” technique for error and success amplification, and analyze it using the Chernoff bound (Thomas Vidick, CMS139 notes).

Now, we set t=O(log⁡1/δ)t = O(\log1/\delta)t=O(log1/δ) and run ttt trials each with failure probability δ=1/5\delta = 1/5δ=1/5, which gives us the value of k=5/ϵ2k = 5/\epsilon^2k=5/ϵ2 (the number of hash functions we need).

Now, let d^1,d^2,⋯ ,d^t\hat{d}_1, \hat{d}_2, \cdots, \hat{d}_td^1​,d^2​,⋯,d^t​ be the outcomes of the ttt trails, which is the estimated value of the number of distinct elements. Then, we set

d^=median(d^1,d^2,⋯ ,d^t) \hat{d} = \mathrm{median}(\hat{d}_1, \hat{d}_2, \cdots, \hat{d}_t) d^=median(d^1​,d^2​,⋯,d^t​)

We know that for the median μ~\tilde{\mu}μ~​ of a random variable XXX are all points xxx such that

Prob(X≤μ~)=Prob(X≥μ~)=12(12) \mathrm{Prob}(X \leq \tilde{\mu}) = \mathrm{Prob}(X \geq \tilde{\mu}) = \frac{1}{2} \tag{12} Prob(X≤μ~​)=Prob(X≥μ~​)=21​(12)

Assume the true value is ddd, and we bound our error with the following range

[(1−4ϵ)d,(1+4ϵ)d] [(1-4\epsilon)d, (1+4\epsilon)d] [(1−4ϵ)d,(1+4ϵ)d]

Then, if more than half of trials fill in the above range, then the equation (12) tells us that the median for sure will fall in the above range. Now, let’s assume that 2/32/32/3 of trials fall in the range, then the median will fall in too.

Since the failure probability δ=1/5\delta = 1/5δ=1/5 , then we know for all ttt trials, each of them can fall in the the error bound with probability at least 4/54/54/5 (that’s the meaning and purpose of failure probability). Now, we want to estimate the probability that the median d^\hat{d}d^ falling in the error bound we set up.

Let random variable XXX be the number of trails of falling in [(1−4ϵ)d,(1+4ϵ)d][(1-4\epsilon)d, (1+4\epsilon)d][(1−4ϵ)d,(1+4ϵ)d], then we can have

The equations (13) means that the probability of median d^\hat{d}d^ does not fall in the range is bounded by Pr(x<23t){Pr}\left ( x < \frac{2}{3} t \right)Pr(x<32​t) for the obvious reason (definition of median).

Since we set up δ=1/5\delta = 1/5δ=1/5, which means that each trail falls in the range with probability at leas 4/54/54/5, therefore we could have

This means the space complexity does not depend on the number of distinct elements ddd or the number of items in the stream nnn (both of these numbers are typically very large).

Algorithms in practice

Right now, our algorithm uses continuous valued fully random hash functions. Howwever, this can’t be implemented. Therefore, we need to extend the analysis into the discrete field.

In practice, it is pretty amazing that we can actually reduce the space complexity down to log⁡log⁡n\log \log nloglogn bits (maybe not that amazing as we are only trying to get zero-moment from one data pass). This algorithm is often referred to as “approximate counting”, “probabilistic counting2, or simply the “LogLog algorithm” (with HyperLogLog being the most recent version (Flajolet et al., 2007) ).
