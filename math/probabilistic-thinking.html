Develop Some Fluency in Probabilistic Thinking

The foundation of machine learning and data science is probability theory. In this post, we will develop some fluency in probabilistic thinking with different examples, which prepare data scientists well for the sexist job of the 21st century.

probability, algorithm, data-science, machine-learning

When I was studying probability theories, it was difficult for me to see its application in real life until we enter a new era with all potential of machine learning and deep learning models. This post is to explain how those basic theories (especially those related to inequalities) could help us for solving real world problems. Personally, I benefit a lot from those examples because it shows me what is probabilistic thinking and how powerful it is. Example 1: verify a claim

Markov’s Inequality

If XXX is a nonnegative random variable and a>0a > 0a>0, then the probability that XXX is at least aaa is at most the expectation of XXX divided by aaa

P(X≥a)≤E[X]a(5) \mathbb{P}(X \geq a) \leq \frac{ \mathrm{E} [X]}{a} \tag{5} P(X≥a)≤aE[X]​(5)

E[X]=∑P(X=x)⋅x≥∑x≥aP(X=x)⋅s≥∑x≥aP(X=x)⋅a=a⋅P(X≥a) \begin{aligned} \mathrm{E}[X] = \sum \mathbb{P}(X = x) \cdot x & \geq \sum_{x \geq a} \mathbb{P}(X = x) \cdot s \\ & \geq \sum_{x \geq a} \mathbb{P}(X = x) \cdot a \\ & = a \cdot \mathbb{P}(X \geq a) \end{aligned} E[X]=∑P(X=x)⋅x​≥x≥a∑​P(X=x)⋅s≥x≥a∑​P(X=x)⋅a=a⋅P(X≥a)​

Example 2: approximate counting

We are using this example to explain the difference between deterministic models and stochastic models, which is relevant for the example 3 in the coming section. This section is also connected with deterministic and stochastic optimization problems.

Approximate Counting algorithms are techniques that allow us to count a large number of events using a very small amount of memory. It was invented by Robert Morris in 1977 and was published through his paper Counting large number of events in small registers. The algorithm uses probabilistic techniques to increment the counter, although it does not guarantee the exactness it does provide a fairly good estimate of the true value while inducing a minimal and yet fairly constant relative error.

Here is the problem:

  • we need to counter a large number, say 1,000,0001,000,0001,000,000
  • however, our register is only 8-bits, which means 28=2562^8 = 25628=256
  • how could count this large number of events and store the value?

The idea is very simple: instead of counting one by one, we could count 1 after 10 times of even or log⁡2(1+n)\log_2(1+n)log2​(1+n). This means we will have a function to map nnn to XnX_nXn​ like the following one:

Xn=log⁡2(1+n) X_n = \log_2(1 + n) Xn​=log2​(1+n)

The tradeoff is to lose accuracy to count a large number :

  • log⁡2(1+0)=0\log_2(1+0) = 0log2​(1+0)=0, we start from 0
  • log⁡2(1+100)=6.64\log_2(1+100) = 6.64log2​(1+100)=6.64, we may count it as 6 or 7
  • log⁡2(1+120)=6.91\log_2(1+120) = 6.91log2​(1+120)=6.91, we may still count it as 6 or 7
  • log⁡2(1+130)=7.02\log_2(1+130) = 7.02log2​(1+130)=7.02, we may now count it as 7

Therefore the accuracy rate of the above algorithm depends on how we round our errors. We could design a rule as follows:

  • round up if value of the first decimal digit is greater or equal than 555
  • round down otherwise

This kind of algorithm is called deterministic as the rule is determined pre-running and the error was also introduced deterministically. However, we could also introduce the error stochastically as a random variable following some distribution.

Example 3: hash tables

This section assumes readers know what hash table is. If not, just check out this video from MIT.

The hash function h:U→[n]h: U \to [n]h:U→[n] maps the elements of universe or population to indices 1,⋯ ,n1, \cdots, n1,⋯,n of an array. The data structure dict from Python is built on hash tables as this kind of map makes it easy to extract information.

For a big dataset of ∣U∣>>n|U| >> n∣U∣>>n, we need to reply on probability to map elements from universe to the array as the it is economically efficient. For instance, big firms like Google or Facebook are always trying to use less servers or database to serve their customs. In this case the population size is much larger than the number of servers or database.

Figure 4. Illustration of hash table for IP addresses, figure was taken from Musco's course (2022).

As we have explained that it costs a lot to have an index for each element shown in Figure 4. To use less indexes in our hash table, we need to find a way to avoid collisions.

Now, we store mmm items from a large universe in a hash table with nnn positions. When we insert mmm items into the hash table we may have to store multiple items in the same location (typically as a linked list).

Let’s frame our problem as a probabilistic one:

  • mmm: total number of stored items (sample size)
  • nnn: hash table size (population size)
  • xi,xjx_i, x_jxi​,xj​: any pair of store items chosen randomly
  • CCC: total pairwise collisions
  • hhh: random hash table function

Remark: In example 1, the population size n=1,000,000n=1,000,000n=1,000,000 is fixed, whereas the sample size mmm is updated dynamically every time we draw a sample; for example 2, the hash table size nnn is fixed too, whereas total number of stored items is updated dynamically. It is important to know when and how to set up population and sample space. It might be counterintuitive when you see we set the hash table as the population.

General principle for setting up population and sample: Anything is fixed should be treated as the population, whereas anything is updated dynamically should be treated as the sample space that could be mapped with the random variable function.

With the right setup, we can have the following expectation of pairwise collision

E[C]=∑i,j∈[m],i<jE[Ci,j]=(m2)1n=m(m−1)2n(6) \mathrm{E} [C] = \sum_{i, j \in [m], i < j} \mathrm{E} [C_{i, j}]= \binom{m}{2} \frac{1}{n} = \frac{m(m-1)}{2n} \tag{6} E[C]=i,j∈[m],i<j∑​E[Ci,j​]=(2m​)n1​=2nm(m−1)​(6)

For n=4m2n = 4m^2n=4m2 (size of hashing table is much larger than the sample size), we have

E[C]=m(m−1)8m2<m28m2=18(7) E[C] = \frac{m(m-1)}{8m^2} < \frac{m^2}{8m^2} = \frac{1}{8} \tag{7} E[C]=8m2m(m−1)​<8m2m2​=81​(7)

This means we could have the collision-free hash table if the hash table is large enough. This completely makes sense. However, we do not gain any economic benefit as we are using O(m2)O(m^2)O(m2) space to store mmm elements.
